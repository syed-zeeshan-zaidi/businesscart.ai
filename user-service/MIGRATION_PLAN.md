# User-Service Migration Plan: TypeScript to Go

This document outlines the plan for migrating the `user-service` from a TypeScript and Node.js-based AWS Lambda function to a Go-based implementation.

## 1. Current Implementation (TypeScript)

### 1.1. Overview

The current `user-service` is a serverless application built with TypeScript, Node.js, and the AWS CDK. It manages user registration, authentication, and basic user management. It uses MongoDB as its database.

### 1.2. Database Schema

The service uses a MongoDB database named `UserService` with the following collections:

*   **`users`**:
    *   `name`: `String` (required)
    *   `email`: `String` (required, unique)
    *   `password`: `String` (required, hashed with bcrypt)
    *   `role`: `String` (enum: `customer`, `company`, `admin`, required)
    *   `company_id`: `String` (optional, for company users)
    *   `associate_company_ids`: `[String]` (optional, for customer users)
    *   `createdAt`: `Date`
    *   `updatedAt`: `Date`
*   **`refreshtokens`**:
    *   `userId`: `ObjectId` (ref: `User`, required)
    *   `token`: `String` (required, unique)
    *   `expiresAt`: `Date` (required)
*   **`blacklistedtokens`**:
    *   `token`: `String` (required, unique)
    *   `expiresAt`: `Date` (required)
*   **`businesscodes`**:
    *   `business_code`: `String` (required, unique)
    *   `company_access_code`: `String` (required, unique)
    *   `company_id`: `String` (required, unique)

### 1.4. Authentication Flow

1.  **Registration**: A new user is created. `company` and `customer` roles require a `business_code` or `company_access_code` respectively, which is validated against the `businesscodes` collection.
2.  **Login**: The user provides an email and password. The credentials are validated, and a short-lived JWT access token and a long-lived refresh token are generated and returned.
3.  **Authenticated Requests**: The access token is sent in the `Authorization` header for protected endpoints.
4.  **Token Refresh**: When the access token expires, the client sends the refresh token to the `/users/refresh` endpoint to get a new access token.
5.  **Logout**: The client sends both tokens to the `/users/logout` endpoint. The access token is added to a blacklist, and the refresh token is deleted.

### 1.5. CDK Infrastructure (`lib/user-service-stack.ts`)

*   A `lambda.Function` is defined with the Node.js 18.x runtime.
*   The code is loaded from the `user-service/dist` directory.
*   Environment variables (`MONGO_URI`, `JWT_SECRET`, `JWT_REFRESH_SECRET`, `NODE_ENV`) are passed to the Lambda function from a `.env` file.
*   An `apigateway.LambdaRestApi` is created to expose the Lambda function.
*   Specific resources and methods are added to the API Gateway to route requests to the single Lambda handler.

## 2. Proposed Implementation (Go)

### 2.1. Overview

The new service will be a Go application compiled into a single binary and deployed as an AWS Lambda function. It will maintain full API compatibility with the TypeScript version.

       config.go
    handler/            // HTTP handlers
       http.go
    storage/            // Database interaction
        models.go       // Go structs for DB models
